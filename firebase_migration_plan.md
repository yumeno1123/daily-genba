# Firebase 移行・共有機能実装計画書

このドキュメントは、「daily-genba」アプリを個人専用からチーム全員で共有・リアルタイム同期できるアプリへアップグレードするための計画をまとめたものです。

---

## 1. 移行の目的
- **リアルタイム共有**: 現場で入力した日報を、事務所のPCや他のメンバーのスマホで即座に確認可能にする。
- **データバックアップ**: ブラウザのキャッシュ削除やデバイスの故障によるデータ消失を防ぎ、クラウド上で安全に管理する。
- **マルチデバイス対応**: 外出先ではスマホ、事務所ではPCといった使い分けをスムーズにする。

---

## 2. 導入フェーズと具体的な手順

### 【フェーズ 1】 Firebase プロジェクトの準備
1. **プロジェクト作成**: [Firebase Console](https://console.firebase.google.com/) で新規プロジェクト `daily-genba` を作成。
2. **Firestore の有効化**: 
    - 「Cloud Firestore」を選択し、データベースを作成。
    - ロケーションを `asia-northeast1` (東京) に設定。
    - 初期ルールを「テストモード」で開始（誰でも読み書き可能な状態）。
3. **アプリの登録**: 
    - Webアプリとして登録し、`firebaseConfig`（APIキー等）を取得。

### 【フェーズ 2】 アプリの通信基盤の書き換え
1. **ライブラリのインストール**: 
    - `npm install firebase` を実行。
2. **`services/db.ts` の刷新**:
    - `idb` ライブラリの依存を削除し、Firebase Firestore SDK を導入。
    - 現場（Projects）と日報（DailyRecords）の各関数（get, add, update, delete）を Firebase 用に書き換え。
3. **ID形式の変更**:
    - IndexedDB の「数字ID (1, 2, 3...)」から Firebase の「ランダム文字列ID」へ対応。
    - プログラム内の `parseInt` や `number` 型の ID 処理を `string` へ修正。

### 【フェーズ 3】 リアルタイム同期とUIの調整
1. **リアルタイム監視の実装**:
    - `getDailyRecords` などを、1回限りの取得から `onSnapshot`（変更を常に監視）による自動更新に変更。
2. **オフライン対応の強化**:
    - Firebase の「オフライン・パーシスタンス」機能を有効化し、電波のない現場でも入力可能にする（電波復活時に自動送信）。

### 【フェーズ 4】 既存データの移行（マイグレーション）
1. **移行ツールの作成**: 
    - 現在のブラウザ（IndexedDB）から全データを読み取り、Firebase へ一括アップロードするボタンを一時的に実装。
2. **データのクリーンアップ**:
    - 移行完了後、IndexedDB をクリアしてクラウド運用へ完全移行。

---

## 3. Firebase 無料枠（Sparkプラン）の制限と運用コスト
この計画は、Google が提供する無料枠の範囲内で十分に運用可能です。

- **読み取り**: 50,000回 / 日
- **書き込み**: 20,000回 / 日
- **削除**: 20,000回 / 日
- **保存容量**: 1 GiB まで
- **通信量**: 10 GiB / 月 まで

※ 10人程度のチームで毎日数件の日報を登録する程度であれば、追加費用が発生することはありません。

---

## 4. 期待される効果
1. **事務所の手間削減**: 日報を回収・集計する作業が不要になり、CSV出力も事務所で即座に行える。
2. **情報の透明化**: 今どの現場に誰が入っているかが、アプリを開くだけで全員に共有される。
3. **安心感**: 携帯を紛失しても、新しい端末でアクセスすればデータが失われない。
